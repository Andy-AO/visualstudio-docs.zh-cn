# <a name="c26117"></a>C26117
警告 C26117:解除未持有的锁\<锁 > 函数中\<f u n c >。

 语法上限定了作用域锁强制*获取*锁*发行*C 中的对 /C++程序不由语言。 通过对并发状态进行显著的修改，函数可能会造成锁定副作用。 例如，锁包装器函数递增的锁获取或锁计数，给定锁的数目。可以对具有副作用锁获取或锁释放使用的函数进行批注`_Acquires_lock_`或`_Releases_lock_`分别。 如果没有此类批注，函数在返回之后，预计不会更改任何锁计数。 如果获取和发布不平衡，则它们被视为*孤立*。 在不使用批注的函数时才发出警告 C26117`_Releases_lock_`释放锁不会保留，因为该函数必须拥有该锁之前会释放。

## <a name="example"></a>示例
 下面的示例生成警告 C26117，因为函数`ReleaseUnheldLock`释放锁，它不一定会保存 — 的状态`flag`是不明确的并且没有指定它应没有任何批注。

```cpp

typedef struct _DATA
{
    CRITICAL_SECTION cs;
} DATA;

int flag;

void ReleaseUnheldLock(DATA* p)
{
    if (flag)
        EnterCriticalSection(&p->cs);
    // code ...
    LeaveCriticalSection(&p->cs);
}
```

## <a name="example"></a>示例
 下面的代码修复问题，通过确保在相同情况下还获得了释放的锁。

```cpp

typedef struct _DATA
{
    CRITICAL_SECTION cs;
} DATA;

int flag;

void ReleaseUnheldLock(DATA* p)
{
    if (flag)
    {
        EnterCriticalSection(&p->cs);
        // code ...
        LeaveCriticalSection(&p->cs);
    }
}
```
