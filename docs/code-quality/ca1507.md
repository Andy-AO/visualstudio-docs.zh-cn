# <a name="ca1507-use-nameof-in-place-of-string"></a>CA1507：使用 `nameof` 替换字符串

|||
|-|-|
|TypeName|UseNameofInPlaceOfString|
|CheckId|CA1507|
|类别|Microsoft 可维护性|
|重大更改|不间断|

## <a name="cause"></a>原因

与包含方法的参数名称或包含类型的属性名称相匹配的 @no__t 0 文本或常量，用作方法的参数。

## <a name="rule-description"></a>规则说明

规则 CA1507 将 `string` 文本标记为方法或构造函数的参数，其中[nameof](/dotnet/csharp/language-reference/keywords/nameof) （@no__t 为 Visual Basic）表达式中的可添加可维护性。 如果满足以下所有条件，则会触发规则：

- 参数是 @no__t 0 文本或常数。

- @No__t 的常量值与以下两项之一匹配：

  - 要在其中调用方法或构造函数的方法、lambda 或本地函数的参数的名称。

  - 要在其中调用方法或构造函数的类型的属性的名称。

- 参数对应于方法的 @no__t 旁1/-0 类型参数或正在被调用的构造函数（即，调用站点不涉及转换）。

规则 CA1507 可提高代码的可维护性，这种情况下，可能会在将来重命名参数，但未将 `string` 文本错误地重命名。 通过使用 `nameof`，在通过重构操作重命名该参数时，将重命名符号。 此外，编译器将捕获参数名称中的任何拼写错误。

> [!NOTE]
> 规则 CA1507 在旧版分析中不可用。 它最初是在[FxCop 分析器](https://www.nuget.org/packages/Microsoft.CodeAnalysis.FxCopAnalyzers)版本2.6.0 中引入的。

## <a name="how-to-fix-violations"></a>如何解决冲突

若要解决冲突，请将 @no__t 0 文本替换为[nameof](/dotnet/csharp/language-reference/keywords/nameof) （在 Visual Basic 中为 @no__t 2）表达式。 例如，以下两个代码段显示了规则冲突以及如何修复此问题：

```csharp
public Book(string title)
{
    // Violates rule CA1507
    Title = title ?? throw new ArgumentNullException("title", "All books must have a title.");
}
```

```csharp
public Book(string title)
{
    // Resolves rule CA1507 violation
    Title = title ?? throw new ArgumentNullException(nameof(title), "All books must have a title.");
}
```

> [!TIP]
> 在 Visual Studio 中，可以为此规则提供代码修补程序。 若要使用它，请将光标置于 @no__t 0 文本上，然后按**Ctrl**+ **。** （句点）。 选择 "使用 nameof" 以从显示的选项列表中**表达符号名称**。
>
> ![CA1507 的代码修复-使用 nameof 表达符号名称](media/ca1507-code-fix.PNG)

## <a name="when-to-suppress-warnings"></a>何时禁止显示警告

如果你不关心代码的可维护性，则可以安全地禁止违反此规则。
