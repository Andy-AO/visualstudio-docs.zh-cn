# <a name="c26116"></a>C26116
警告 C26116：未能获取或 @no__t > \<func > 中的锁。

 C/C++ program 中的语义范围内锁*获取*和锁*释放*对不是由语言执行的。 通过对并发状态进行显著的修改，函数可能会造成锁定副作用。 例如，锁包装函数增加了给定锁的锁获取数或锁计数。您可以通过分别使用 @no__t 0 或 @no__t，批注具有从锁获取或锁释放产生副作用的函数。 如果没有此类批注，函数在返回之后，预计不会更改任何锁计数。 如果获取和释放不平衡，它们将被视为*孤立*。 当函数已使用 `_Acquires_lock_` 进行批注时，将发出警告 C26116，但它不获取锁，或者当函数使用 @no__t

## <a name="example"></a>示例
 下面的示例将生成警告 C26116，因为 @no__t 的函数 `DoesNotLock` 的批注，但没有获取它。 函数 `DoesNotHoldLock` 会生成警告，因为它是用 @no__t 批注的，不包含它。

```

typedef struct _DATA
{
    CRITICAL_SECTION cs;
} DATA;

_Acquires_lock_(p->cs) void DoesLock(DATA* p)
{
    EnterCriticalSection(&p->cs); // OK
}

_Acquires_lock_(p->cs) void DoesNotLock(DATA* p)
{
    // Warning C26116
}

_Requires_lock_held_(p->cs) void DoesNotHoldLock(DATA* p)
{
    LeaveCriticalSection(&p->cs); // Warning C26116
}
```
